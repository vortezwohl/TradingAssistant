## Context

当前页面已经具备可工作的 bootstrap、WebSocket 与指标摘要链路，但布局仍停留在“标题 + 左侧控制栏 + 右侧图表卡片”的后台样式。`tradingassistant/frontend/app.py` 目前在首屏同时展示标的选择、周期选择、指标开关、连接信息和图表工作区；`tradingassistant/frontend/charting.py` 则延续亮底图表与浅色网格的默认视觉。这与本项目已经具备的实时图表能力不匹配，也无法体现交易型界面应有的阅读顺序：先看图，再做切换，最后再看低频设置与诊断。

本次设计属于跨模块前端重构：既涉及页面骨架，也涉及图表容器、摘要面板、响应式行为和主题变量。实现时必须保持现有数据链路与状态边界不变，继续遵守“高频运行态留在浏览器，不回灌 Reflex State”的既有原则。

## Goals / Non-Goals

**Goals:**

- 将看盘页重构为图表优先的交易工作区，使首屏视觉中心稳定落在主图区域。
- 建立统一的暗冷主题令牌，覆盖页面背景、面板、边框、文字、强调色和状态色。
- 把低频设置与诊断信息改造成渐进披露结构，避免它们与主图争抢首屏注意力。
- 在不修改后端协议的前提下，保留并强化 forming/closed bar 与 provisional/finalized 指标的 UI 提示。
- 定义桌面、平板、移动端下的稳定布局切换与滚动策略，确保页面更像 WebApp 而不是工业监控台。

**Non-Goals:**

- 这次设计不引入新的后端接口、订阅协议或指标计算逻辑。
- 这次设计不实现多窗口联动、画线工具、模板系统或桌面终端级多工作区。
- 这次设计不新增独立 Node/NPM 前端工程，仍保持现有 Reflex 工程边界。
- 这次设计不试图在一个视口内展示所有信息，也不追求把全部低频配置做成常驻面板。

## Decisions

### 1. 采用“页面主题令牌 + 组件级样式助手”的前端组织方式

**Decision**

在 `tradingassistant/frontend/` 内新增或抽取一组主题令牌与布局样式助手，集中定义暗色背景、冷灰边框、橙色强调、状态色、阴影、圆角和间距，而不是把颜色与间距硬编码散落在 `app.py` 和 `charting.py` 中。

**Rationale**

- 当前页面已经出现背景、卡片、badge、图表颜色各自为政的问题，继续在原位追加 style 会快速失控。
- 主题令牌能让页面容器、侧栏摘要、ECharts 配色和状态标签共享同一视觉语言。
- 这种方式仍保持 Python 主导，不需要切换到单独 CSS 工程。

**Alternatives considered**

- 方案 A：直接在现有组件上逐个改颜色。
  - 放弃原因：短期快，但后续很难保证一致性，且不利于继续扩展更多页面。
- 方案 B：一次性引入完整设计系统或独立样式框架。
  - 放弃原因：对当前任务过重，会扩大改动范围。

### 2. 首屏重组为“顶部命令条 + 主图工作区 + 右侧摘要栏”

**Decision**

桌面端首屏采用三段式结构：顶部命令条承载产品名、当前标的、周期切换与关键状态；中部主区域以图表为中心；右侧保留窄摘要栏，只展示当前真正需要扫读的指标与状态。低频设置和开发诊断信息不再常驻首屏。

**Rationale**

- 这符合交易界面的主任务顺序，先看价格与结构，再做切换和判断。
- 右侧摘要栏可以保留 RSI、MACD、通道状态等“次级但高频查看”的信息，不必挤占主图画布。
- 顶部命令条比左侧大控制栏更适合承载低频切换动作，也更接近 Bloomberg 风格的信息条组织方式。

**Alternatives considered**

- 方案 A：保留当前左右双栏，只做暗色换肤。
  - 放弃原因：这只能修表皮，无法解决首屏信息过载的问题。
- 方案 B：把全部控制都做成右侧抽屉。
  - 放弃原因：会让常用切换动作变深，降低首屏操作效率。

### 3. 低频配置与诊断信息采用渐进披露，而非首屏常驻

**Decision**

指标开关、自选列表扩展项、API/WS 连接信息和订阅意图应放入第二屏分区、折叠面板或抽屉；默认状态下只显示与当前判断直接相关的最小摘要，开发诊断面板默认关闭。

**Rationale**

- 这些信息存在真实价值，但大多不属于用户每次进入页面的首要关注点。
- 渐进披露能保留功能完整性，同时显著降低首屏噪声。
- 对联调和开发仍然友好，因为诊断区不是被删掉，而是被下沉和折叠。

**Alternatives considered**

- 方案 A：彻底删除诊断信息。
  - 放弃原因：会损失当前联调阶段的重要可观测性。
- 方案 B：继续常驻首屏但缩小字号。
  - 放弃原因：只是把噪声缩小，没有改变信息架构。

### 4. 图表与摘要面板保持现有数据语义，但切换为暗色专业视觉

**Decision**

继续使用现有浏览器图表运行时结构，不改 bootstrap 和增量处理逻辑；只调整图表容器样式、ECharts 颜色映射、网格线、tooltip、坐标轴、badge 与摘要面板的展示位置和视觉层级。forming/closed 与 provisional/finalized 的语义继续保留，但从“散乱文本”变成更明确的状态标签和摘要文案。

**Rationale**

- 当前的核心问题在于表达方式，不在于数据契约本身。
- 维持现有运行时脚本可以把风险集中在展示层，避免把一次 UI 改版扩散成数据链路改造。
- 暗色主题需要图表与页面统一调整，否则会出现“页面暗、图表亮”的割裂感。

**Alternatives considered**

- 方案 A：趁机重写图表运行时。
  - 放弃原因：超出本次重构边界，回归风险高。
- 方案 B：只改页面容器，不改图表 option。
  - 放弃原因：图表会与新的暗色工作区严重失配。

### 5. 响应式策略采用“允许滚动的单主任务布局”

**Decision**

桌面端采用主图 + 摘要栏并保持二级内容下沉；平板端改为主图在上、摘要与控制在下；移动端采用单列堆叠和可滚动布局，并保留顶部命令条或其简化版本。任何断点下都不要求把所有模块挤进单视口。

**Rationale**

- 用户已经明确指出页面应是 WebApp，而非工业控制软件；滚动是合理而必要的交互成本。
- 交易图表在小屏上首先需要足够可读的主图空间，而不是并列更多卡片。
- 这能降低移动端样式 hack 的复杂度，也更符合现有 Reflex 组件体系。

**Alternatives considered**

- 方案 A：所有断点都维持同一网格结构。
  - 放弃原因：在窄屏下会迅速变成挤压式信息墙。

## Risks / Trade-offs

- [页面重构跨度较大，容易引入样式回归] → 通过主题令牌集中管理样式，并分阶段验证桌面端和移动端布局。
- [暗色主题下图表与表单控件可读性下降] → 单独校验文本、边框、网格线和 tooltip 的对比度，避免只追求气质而牺牲可读性。
- [诊断信息下沉后影响联调效率] → 保留一个默认折叠的 diagnostics 分区，并在开发环境中保持可快速展开。
- [首屏改成图表优先后，部分切换操作路径变长] → 把标的和周期保留在顶部命令条，确保常用动作仍是一跳可达。
- [Reflex 组件样式能力限制视觉表达] → 优先使用可稳定落地的样式属性和现有组件组合，不为追求过度拟真而引入额外框架。

## Migration Plan

1. 先抽取主题令牌与页面骨架辅助函数，避免在旧布局上直接堆叠样式变更。
2. 重建 `app.py` 的页面结构，把首屏切换为顶部命令条、主图工作区和摘要栏，并将诊断区下沉。
3. 调整 `charting.py` 中图表容器、摘要面板和 ECharts 配色，使其与新主题一致。
4. 为桌面端、平板端、移动端补齐断点行为与滚动验证，确认首屏不再过载。
5. 通过本地运行和手工验收确认：图表仍正常接收 bootstrap / WebSocket 更新，视觉和交互完成重构后再继续下一步功能增强。

## Open Questions

- 是否需要在首版中引入更具辨识度的数字字体或等宽字体组合；如果要引入，是否接受新增 Web 字体依赖。
- diagnostics 分区是否只在开发环境显示，还是在生产环境也保留入口。
- 右侧摘要栏是否需要继续显示全部 MA/MACD/RSI 数值，还是首版先压缩成更少但更关键的摘要项。
