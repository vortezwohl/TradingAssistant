TradingAssistant 增强版看盘系统架构设计

面向“类似同花顺的看盘 WebApp”目标，结合 Reflex、FastAPI、iTick、 OpenTrade 的能力边界，设计一套适合实时分时图、实时 K 线、多指标增量计算与多标的订阅分发的增强版系统架构。

生成日期：2026-06-07（Asia/Hong_Kong） | 研究范围：官方文档、设计模式文章、项目 .venv 内依赖源码入口 | 文档状态：V1，适合进入实现前评审

1. 任务定约

Objective
  • 设计一个增强版看盘系统架构，而不是停留在 MVP。
  • 支持实时分时图、实时 K 线、多周期聚合、全量指标计算和多页面订阅分发。
  • 为后续基于 Python 闭环的实现提供清晰模块边界与演化路径。
Non-goals
  • 本阶段不实现业务代码，不定义最终 UI 细节。
  • 本阶段不解决生产环境授权、商业分发许可和运维容量采购问题。
  • 本阶段不把所有指标都写成最终代码，只定义指标引擎策略。
Constraints
  • 前端框架固定为 Reflex，不把 Node/NPM 作为应用开发主链。
  • 行情主数据源固定为 iTick。
  • 指标能力允许用 OpenTrade 做补充。
  • 服务门面固定为 FastAPI。
Verification
  • 架构必须解释实时链路如何闭环。
  • 架构必须说明哪些模式被用于隔离哪些变化点。
  • 架构必须能映射到当前依赖的真实 API 契约，而不是抽象空谈。

2. 研究结论摘要

iTick

  • itick-sdk 0.1.0 的 Python 封装很薄，核心就是 REST + WebSocket 客户端。
  • WebSocket 客户端内置心跳、重连、读线程，但属于线程模型，不是 asyncio 模型。
  • 说明它适合作为“上游接入适配层”，不适合直接成为系统的统一事件总线。

OpenTrade

  • OpenTrade 更强在指标增强、观察输出和多 backend 门面，而不是实时流处理。
  • opentrade.enrichment.indicators 直接把大量指标附着到历史 DataFrame 上。
  • 它适合做“指标公式库 + 离线校验器 + 辅助观察层”，不适合做高频增量计算总线。

Reflex

  • Reflex 0.9.4 本质是带状态管理和 WebSocket 事件处理的前后端一体框架。
  • 内部有 StateManager、事件队列，以及 MEMORY / DISK / REDIS 三种状态管理模式。
  • 说明它适合承载页面状态和交互；当前方案默认走 MEMORY，不承接所有逐笔行情事件。

FastAPI

  • FastAPI 适合作为系统门面、统一 WebSocket / SSE / REST 出口。
  • 它应成为“行情内核”和“Reflex 页面层”之间的协议边界。
  • 这能避免高频实时链路与页面状态机耦死。

3. 设计模式选型与变化点映射

参考《AI Coding 时代，理解设计模式变得更加重要》中的核心观点，本系统优先围绕“变化点”选模式，而不是为了模式而模式。 实时行情系统的主要变化点包括：上游数据源协议、分发通道、K 线聚合算法、指标计算算法、图表展示方式、会话级订阅管理。

模式 隔离的变化点 在本系统中的落点 为什么必要
Adapter iTick REST / WebSocket 的原始协议、字段风格、线程模型 MarketDataAdapter、SymbolAdapter、QuoteNormalizer 避免上游字段和连接策略直接污染领域层与页面层。
Facade 页面不应了解订阅、回填、指标、聚合的编排细节 MarketFacade / ChartFacade 把“打开一张实时图表”收敛成单一入口。
Observer 行情到达后有哪些消费者需要响应 事件总线、主题订阅、图表广播、预警监听 行情源不应该知道有多少页面、多少指标、多少下游处理器。
Strategy K 线聚合规则、指标计算规则、推送节流策略 BarAggregationStrategy、IndicatorStrategy、PushPolicyStrategy 算法会持续变化，不能长成条件泥潭。
Factory Method 不同市场、不同周期、不同图表上下文的处理器创建 ProcessorFactory、IndicatorEngineFactory 创建逻辑集中，扩市场和扩周期时影响面更小。
Proxy 缓存、权限、限流、懒加载、降级访问 CachedHistoryProxy、RateLimitProxy 保证页面请求不会直接穿透到高成本上游调用。
Template Method “回填历史 → 订阅实时 → 聚合 → 计算指标 → 广播”骨架 RealtimePipelineTemplate 主流程固定，局部步骤允许替换。
本系统明确不推荐滥用 Singleton。全局注册表、配置门面可以是单例语义， 但可变业务状态、订阅上下文、会话状态不应偷懒做成全局对象。

4. 总体架构

增强版目标下，必须把系统拆成“页面层”“门面层”“行情内核层”“计算层”“缓存与分发层”五块。 核心原则是：让 Reflex 管页面，让 FastAPI 管协议，让行情内核管实时状态。

5. 关键实时流程

关键判断：分时图和实时 K 线的“初始化”与“增量更新”必须拆开。 初始化阶段允许使用 OpenTrade 做全量 DataFrame 指标补齐， 但实时阶段必须转为增量策略，否则计算成本会随着窗口长度和订阅数线性放大。

6. 模块划分

market_gateway

  • 封装 iTick REST / WebSocket 接入。
  • 屏蔽线程模型、心跳、重连、原始字段差异。
  • 统一输出标准化事件：TickEvent、QuoteEvent、KlineEvent。

subscription_hub

  • 管理 symbol + period + topic 与 session 的映射。
  • 实现“一个上游订阅，多前端共享”的扇出。
  • 负责无人订阅时延迟退订。

history_service

  • 封装历史 K 线回填与缓存。
  • 当前优先命中进程内 MEMORY 缓存，其次调用 iTick REST。
  • 为指标初始化提供稳定基线数据。

bar_engine

  • 维护正在形成的 1m bar。
  • 以 1m 为基础聚合出 5m / 15m / 30m / 60m / 日线。
  • 处理闭市、缺口、停牌、跨交易段问题。

indicator_engine

  • 分为初始化器、增量计算器、校验器三层。
  • 初始化器可复用 OpenTrade DataFrame 指标。
  • 增量计算器用状态对象承载 EMA、RSI、MACD、BOLL 等递推数据。

push_gateway

  • 对外暴露 FastAPI WebSocket / SSE。
  • 根据前端能力决定推送粒度。
  • 支持图表补点、指标补点、列表快照分通道下发。

reflex_app

  • 负责页面布局、选股、图层切换、指标开关。
  • 保留用户交互状态、自选列表、窗口布局等低频状态。
  • 不负责承载所有逐笔行情状态。

alert_center

  • 监听指标交叉、阈值突破、异动事件。
  • 采用 Observer 模式与主链路解耦。
  • 后续可扩展站内提醒、Webhook、策略信号。

ops_support

  • 指标计算耗时、每主题订阅数、上游连接健康度。
  • 缓存命中率、延迟分位数、丢包率、重连次数。
  • 为增强版进入生产前提供可观测性。

7. 实时图表设计

分时图

  • 基线数据：使用当日分时历史快照或当日 1m 数据回填。
  • 实时驱动：优先用 tick，次选 quote。
  • 衍生线：均价线、涨跌幅线、成交量柱、成交额柱。
  • 推送策略：高频补点，低频重算窗口统计。

实时 K 线

  • 首屏用 REST K 线回填。
  • 实时层维护 forming bar，前端显示“未闭合 K 线”。
  • 多周期一律从 1m 或更细粒度聚合，避免上游多路重复订阅。
  • 指标分为“形成中指标”和“闭合后正式指标”两套视图。

8. 指标引擎设计

层次 职责 建议实现 示例指标
初始化层 对历史窗口一次性补齐指标列 调用 opentrade.enrichment.indicators MA、EMA、MACD、RSI、KDJ、BOLL、ATR、VWAP、SuperTrend
增量层 每次新 tick / 新 bar 到达时最小代价更新 自定义状态对象 + Strategy EMA、MACD、RSI、OBV、ATR、VWAP、PVT
窗口层 维护固定窗口、极值与统计摘要 Ring Buffer / Deque / 单调队列 BOLL、Donchian、CCI、WR、KDJ
校验层 验证增量算法与全量算法的一致性 定时抽样重跑 OpenTrade 随机标的随机周期一致性比对

指标实现原则

  • 同一指标必须定义统一字段名和统一精度规则，避免前后端名称漂移。
  • 形成中的 bar 可以计算预估指标，但必须显式标记为 provisional。
  • 闭合后的 bar 才写入正式缓存，并触发预警判断和高周期聚合。
  • OpenTrade 在本项目中的最佳角色是“初始化器 + 校验器 + 指标覆盖补充源”。

9. Reflex 与 FastAPI 的边界

Reflex 适合承载

  • 页面路由、布局、筛选条件、用户交互。
  • 自选分组、当前选中标的、当前选中周期、当前开启指标集合。
  • 低频业务状态和用户会话偏好。

Reflex 不适合承载

  • 每个 symbol 的逐笔 tick 状态。
  • 所有图表全量数据的 server-side state 镜像。
  • 高频到达的原始实时数据缓冲。
原因很直接：Reflex 内部有状态序列化和事件处理队列；虽然它支持 MEMORY / DISK / REDIS， 但本项目默认把高频实时状态放在行情内核，不放进 Reflex State。 如果把高频行情直接灌入 Reflex State，会把页面状态管理问题和实时数据流问题硬绑到一起， 后续扩标的、扩多页面、扩共享订阅时成本会迅速上升。

10. 缓存与分发策略

L1：进程内热缓存

  • 保存热门 symbol 的最新 quote、forming 1m bar、最近 N 根 bar。
  • 目标：低延迟，作为当前版本的主缓存层。

L2：Redis 预留层

  • 当前不启用，但接口上预留共享行情快照、主题订阅计数、历史窗口缓存、跨进程广播能力。
  • 目标：未来支撑多 worker、跨实例广播与重连恢复。

L3：上游调用

  • 历史数据和冷 symbol 首次访问时才命中 iTick REST。
  • 目标：控制成本、降低速率限制风险。

广播策略

  • 列表行情和图表行情拆 topic，不让一个页面为另一个页面付出无关流量。
  • 图表 topic 采用 symbol + timeframe + indicator-set-hash 作为二级细分键。
  • 可以把原始上游事件和前端消费事件分离，前者高频，后者经聚合和节流后再推送。

为升级 Redis 预留的抽象边界

  • 缓存访问统一走 CacheStore 接口，先提供 MemoryCacheStore 实现。
  • 主题广播统一走 TopicBus 接口，先提供进程内实现，后续可替换为 Redis Pub/Sub。
  • 订阅注册统一走 SubscriptionRegistry 接口，先使用进程内字典，后续可迁移到 Redis 哈希或集合。
  • 历史窗口快照与形成中 bar 快照都使用统一序列化结构，避免未来迁移时重写协议。
当前策略是“默认 MEMORY，设计上不绑死 MEMORY”。 这意味着业务逻辑不能直接依赖 Python 全局字典，而必须依赖抽象接口访问缓存和广播。

11. 推荐的接口契约

REST

  • GET /api/symbol/search
  • GET /api/chart/bootstrap?symbol=...&period=1m&bars=240
  • GET /api/watchlist
  • GET /api/market/snapshot?symbols=...

WebSocket / SSE

  • /ws/chart：图表增量
  • /ws/quotes：列表行情
  • /ws/alerts：指标事件与异动
  • 消息统一包含 topic、ts、version、payload_type

13. 主要风险与应对

风险：上游数据授权与再分发限制

  • 技术上能做，不代表许可上能公开分发。
  • 应在立项前确认 iTick 的使用范围、商业授权和再分发限制。

风险：把 Reflex 当实时总线

  • 会导致状态臃肿、序列化成本上升、页面刷新过重。
  • 应坚持“Reflex 管页面，FastAPI 管流，内核管状态”。

风险：MEMORY 方案天然不支持横向扩容

  • 单进程方案简单，但无法天然共享跨实例热状态和广播。
  • 应通过 CacheStore / TopicBus / SubscriptionRegistry 抽象，确保后续可平滑升级到 Redis。

风险：指标全量重算

  • 随着订阅数增长会迅速失控。
  • 应尽早投入增量指标状态模型和一致性校验工具。

风险：多周期直接多路订阅

  • 会增加上游连接压力和事件风暴。
  • 应统一以 1m 为基础聚合高周期，必要时再对更细粒度做特例。

14. 本次结论

这个项目要做成“增强版”，正确方向不是把 Reflex、iTick、 OpenTrade、FastAPI 四者直接平铺拼接，而是明确：

  • iTick 是上游数据源与协议接入层。
  • FastAPI 是系统门面与实时分发边界。
  • OpenTrade 是指标初始化与补充校验层，而不是实时流核心。
  • Reflex 是页面组织与交互层，而不是高频行情状态容器。
  • 当前默认运行态存储选择 MEMORY，但缓存、广播、订阅注册均通过抽象接口预留升级到 Redis 的空间。

只要守住这个边界，再用 Adapter、Facade、Observer、Strategy、Proxy、Template Method 去压住变化点，后续实现就能在保持 Python 技术闭环的前提下，逐步逼近专业看盘终端体验。
